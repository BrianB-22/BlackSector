# 1. Purpose

Defines how the Command Line Interface (CLI) client operates using SSH sessions.

**SSH is the primary client interface for Black Sector in v1.** The game is a rich text-based experience accessed via SSH. Players connect using any standard SSH client — no dedicated application is required.

The SSH port (default: 2222) is the designated TEXT mode interface. Connecting to this port automatically establishes a TEXT mode session. A separate TCP port (default: 2223) is reserved for future GUI clients. See `gui_adapter.md`.

In the initial architecture, the SSH session itself functions as the CLI client. The server provides the interactive interface through the TEXT adapter, which renders terminal output and processes keyboard input.

---

# 2. Scope

IN SCOPE:

- SSH-based CLI interaction
- Terminal rendering through TEXT adapter
- Keyboard input handling
- Terminal capability requirements
- SSH session lifecycle

OUT OF SCOPE:

- SSH authentication configuration
- Terminal emulator implementation
- GUI client rendering
- Transport encryption details

---

# 3. Design Principles

The SSH CLI client model follows these principles:

- The SSH session acts as the CLI client.
- The server maintains full control of interface rendering.
- Terminal output is generated by the TEXT adapter.
- Simulation logic remains UI-agnostic.
- SSH transport does not affect protocol semantics.

The simulation engine never emits raw terminal formatting.

All simulation output is first converted into structured messages, which are then rendered by the TEXT adapter.

---

# 4. Architecture Model

The CLI interface operates through the following pipeline:

Simulation Engine  
↓  
Structured Protocol Messages  
↓  
TEXT Adapter  
↓  
ANSI Terminal Rendering  
↓  
SSH Session Terminal  

The SSH session serves as the terminal device for the TEXT adapter.

---

# 5. Terminal Rendering

The TEXT adapter renders structured messages into terminal output using ANSI escape sequences.

Supported terminal capabilities include:

- ANSI color codes
- cursor positioning
- screen clearing
- box drawing characters
- dynamic screen updates

This enables a rich interactive terminal experience.

---

# 6. User Interaction Model

User interaction occurs through keyboard input transmitted via the SSH session.

Examples include:

- command entry
- menu navigation
- hotkeys
- cursor navigation

Input events are interpreted by the server and translated into protocol commands.

Example interaction flow:

User presses key  
↓  
SSH transmits input  
↓  
Server parses command  
↓  
Command submitted to simulation engine  
↓  
Simulation generates events  
↓  
TEXT adapter updates terminal view

---

# 7. Supported SSH Clients

Players may connect using standard SSH clients including:

- OpenSSH
- PuTTY
- Windows Terminal
- iTerm
- SecureCRT

No dedicated client installation is required.

---

# 8. Session Lifecycle

SSH CLI sessions follow this lifecycle:

Connection established on SSH port (2222)
↓
SSH authentication completes
↓
Protocol handshake (TEXT mode implicit — no negotiation)
↓
Terminal interface initialized
↓
Interactive session begins

Session ends when:

- user disconnects
- server terminates session
- connection drops

---

# 9. Terminal Capability Requirements

Recommended terminal capabilities include:

- UTF-8 encoding
- ANSI color support
- cursor control
- minimum 80x24 terminal size

Optional enhancements may include:

- extended color support (256 colors)
- Unicode symbols
- mouse reporting

Terminal capability detection may use environment variables such as:

TERM

Example:

xterm-256color

---

# 10. Integration with Protocol Layer

The CLI interface uses the same protocol message envelope as GUI clients.

TEXT mode operates as an adapter that converts structured protocol messages into ANSI terminal output. The underlying message format is identical.

GUI clients connect on a separate TCP port (2223) and receive the same structured messages without ANSI rendering. See `gui_adapter.md`.

The dual-port model means client type is determined at connection time — no in-session switching.

---

# 11. Advantages of SSH CLI Model

The SSH-based CLI provides several advantages:

- no client installation required
- universal accessibility
- simple debugging and development
- stable protocol interface
- immediate player access

The CLI interface also doubles as a powerful debugging console during development.

---

# 12. Security Considerations

Authentication and encryption are handled by the SSH transport layer.

The protocol layer assumes the connection is already authenticated.

The server must still validate all commands received through the SSH session.

Clients cannot bypass protocol validation.

---

# 13. Non-Goals (v1)

The SSH CLI model does not include:

- embedded terminal emulators
- client-side rendering logic
- browser-based terminal clients
- peer-to-peer CLI sessions

---

# 14. Future Extensions

Possible future improvements include:

- terminal capability negotiation
- enhanced CLI dashboards
- interactive terminal widgets
- CLI scripting support
- multi-pane terminal interfaces

These features must remain separate from the simulation engine.
